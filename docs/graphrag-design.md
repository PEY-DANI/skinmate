# GraphRAG 설계 문서

> 상태: 초안(설계 검토용) · 2026-07-16 · 실데이터(A 신크롤러) 검증 완료

이 문서는 SkinMate의 그래프 기반 추천(GraphRAG) 재설계안을 A(구현 담당)에게 공유하기 위한 것이다.
각 섹션 맨 앞에는 비개발자도 이해할 수 있는 1~2줄 요약을 두고, 이어서 구현에 쓸 수 있는 기술
세부를 정리한다.

---

## 1. 목적과 철학

> **요약**: 지금은 "사용자가 어떤 고민이 있다", "제품에 어떤 성분이 있다"는 사실만 알 뿐,
> "왜 그 고민에 그 성분이 필요한지"를 스스로 설명하지 못한다. 이번 설계는 그 "왜"를 전문
> 아티클에서 뽑아내 근거와 함께 제공하는 것이 목표다.

현재 데이터 모델은 사용자 고민(Concern)과 제품 성분(Ingredient)을 각각 독립적으로 갖고 있지만,
"왜 이 고민에 이 성분이 필요하고, 그래서 왜 이 제품을 추천하는지"를 잇는 연결고리(추천 이유)를
스스로 설명하지 못한다. 아티클(전문 문서)을 분석해 이 연결고리(트리플 관계)를 만들고, 근거
문서와 함께 LLM에 제공함으로써 신뢰도 높은 추천을 생성하는 것이 이번 설계의 목적이다.

핵심 공식:

```
사용자 개인 기억 + 논리적 연결선(트리플 관계) + 실제 근거(아티클 문서) → 신뢰도 높은 추천 콘텐츠
```

---

## 2. 전체 데이터 흐름 (현재 vs 새 설계)

> **요약**: 지금은 검색 방법 세 가지(제품 검색/그래프 탐색/문서 검색)가 서로 남남처럼 따로
> 돈다. 새 설계에서는 질문 속 개념을 먼저 그래프에 연결한 뒤, "이유가 되는 경로"를 찾고, 그
> 경로를 뒷받침하는 문서를 회수하고, 마지막에만 실제 제품을 붙이는 순서로 통합한다.

**왜 필요한가**: 지금은 제품검색·그래프순회·문서검색 세 갈래가 서로 연결되지 않아, "왜 이
성분이 이 고민에 필요한지"라는 논리 연결을 만들 자리가 아예 없다. 그래서 근거 있는 추천(우리
서비스의 차별점)을 못 한다. 이걸 잇는 게 이 설계의 이유다.

### 현재 문제

검색이 세 갈래(제품 벡터검색 / 그래프 순회 / 문서 벡터검색)로 따로 돌고 서로 연결되지 않는다.

- 그래프 순회는 "사용자" 노드에서 출발하는 고정 패턴이다.
- 아티클은 질문과 글자가 비슷한 것만 벡터로 뽑혀 단순 참고자료로만 붙는다.
- 그 결과 "여드름 → 각질제거 → PHA" 같은 논리적 연결을 만들어낼 자리가 애초에 없다.

### 새 흐름

```
질문(한글)
   │
   ▼
① 질문 속 개념 엔티티 링킹
   (여드름 → Concern, 레티놀 → Ingredient)
   │
   ▼
② 그래프에서 메타패스로 "이유" 경로 탐색
   (각 관계 엣지에는 근거 아티클 꼬리표가 붙어 있음)
   │
   ▼
③ 그 관계를 보증하는 아티클 청크 회수
   │
   ▼
④ 개인 기억(memories)으로 필터링/가중치 조정
   │
   ▼
⑤ 실제 제품은 마지막에 RDB에서 JOIN
   (회피 성분은 하드 필터로 제외)
   │
   ▼
LLM 입력 = 개인 기억 + 이유(경로) + 근거 문서 + 제품
   → 근거 있는 추천 생성
```

## 3. 노드 설계 (핵심 — "정체성 키"와 "라벨" 분리)

> **요약**: 그래프 전체에서 언어를 하나로 통일하려 하지 않는다. 노드 종류마다 "그 노드가
> 무엇인지 식별하는 키"와 "화면/질의에 보이는 라벨"을 따로 정한다. 성분은 이미 DB가
> 숫자 ID로 관리하고 있으니 그래프도 그 숫자를 그대로 쓴다.

**왜 필요한가**: canonical_key를 노드 키로 쓰면 한글/영문 표기 문제로 같은 성분이 쪼개지거나
노드가 아예 안 생긴다(Phase 1/2에서 두 번 고쳤는데도 417개가 남음). RDB는 이미 성분을
ingredient_id로 식별하므로(product_ingredients·memories FK), 그래프도 이 안정적 숫자 키를
쓰면 언어 문제 자체가 사라진다.

언어를 전역으로 하나 고르는 대신, 노드 타입별로 정체성(키)과 라벨(매칭용 표시값)을 분리해서
정한다.

| 노드 | 정체성(키) | 라벨·매칭 | 근거 컬럼/출처 |
|---|---|---|---|
| Ingredient | `ingredients.ingredient_id`(숫자 PK, 언어중립) | name_ko(한글) | `ingredients` 테이블. name_en/canonical_key는 보조 |
| Concern | 고정 7종 식별자(영문 enum) | 한글 라벨(트러블 등) | 통제 어휘 시드(CONCERN_RULES name↔label) |
| Mechanism | 한글 canonical 용어(각질제거) | 한글 | `documents.content` 추출. 신설 `concepts` 테이블 권장(id + 한글 용어 + alias) |
| Product | (그래프 밖) `products.product_id` | — | 최종 RDB JOIN에서만 |

성분은 RDB가 이미 `ingredient_id`로 식별하고 있고(`product_ingredients`·`memories`의 FK가
그것을 가리킨다), 그래프도 이 키에 맞춘다. 이렇게 하면 `canonical_key`가 한글이라 노드를 만들지
못하던 문제(417개, §11-2 참고)가 근본적으로 사라진다. Mechanism(작동원리)은 RDB에 대응 테이블이
없고 아티클·질의가 모두 한글이므로 한글 키가 자연스럽다.

## 4. 관계 설계 (6종, 방향 명시)

> **요약**: 그래프 안의 관계는 6가지뿐이다. "성분이 어떤 원리로 작동하는지", "그 원리가 고민에
> 도움/악화가 되는지", "성분끼리 궁합이 맞는지" 세 축으로 나뉜다.

**왜 필요한가**: 관계를 3축으로 좁히면 의미가 명확하고 추출이 안정된다. 정방향만 추출하고
순회에서 역방향을 허용하는 이유는, 아티클은 "성분이 B를 한다"는 정방향 서술이 압도적이고
역방향 독립 문장은 드물어(데이터 6건), 양방향을 억지로 추출하면 품질이 떨어지기 때문이다.

| 관계 | 방향 | 의미 |
|---|---|---|
| ACHIEVES | 성분 → 작동원리 | "이 성분이 이 작동원리를 한다" |
| ENABLES | 작동원리 → 작동원리 | 기전 연쇄 |
| TREATS | (작동원리\|성분) → 고민 | "도움된다" |
| AGGRAVATES | (작동원리\|성분) → 고민 | "악화시킨다" |
| HELPS | 성분 ↔ 성분 | 궁합 좋음(W5용) |
| CONFLICTS | 성분 ↔ 성분 | 궁합 나쁨(W5용) |

추출은 정방향만 수행하고, 그래프 순회 시 역방향 탐색은 허용한다(양방향 추출을 억지로 만들지
않는다).

## 5. 메타패스 화이트리스트

> **요약**: 그래프를 아무 경로로나 헤매지 않도록, 질문에 답할 때 쓸 수 있는 "탐색 경로 패턴"을
> 미리 정해둔다. 질문 속 고민에서 시작해 역방향으로 거슬러 올라가며 읽는다.

**왜 필요한가**: 무제한 다중 hop을 허용하면 두 가지가 깨진다. ① **의미 왜곡** — 예:
"여드름←(도움됨)←살리실릭애씨드→(충돌)→레티놀→(궁합)→히알루론산" 체인을 따라가 "여드름에
히알루론산 제품 추천"을 하면 헛소리다(충돌 관계가 의미를 뒤집는다). ② **속도 폭발** — 촘촘한
그래프에서 hop이 늘수록 경로가 조합적으로 폭증해 응답이 느려진다. 그래서 의미가 이어지는 타입
시퀀스만 화이트리스트로 허용하고 hop을 3으로 제한한다. 신뢰도는 경로 길이가 아니라 근거 문서
수(확증)로 얻는다.

질문 속 고민에서 역방향으로 읽는 방식으로 정의한다.

| 패턴 | 경로 | 설명 |
|---|---|---|
| W1 | 고민 ← 도움됨 ← 작동원리 ← 수행 ← 성분 | 핵심, 2-hop |
| W1' | 고민 ← 도움됨 ← 작동원리 ← enable ← 작동원리 ← 수행 ← 성분 | 3-hop, 기전 연쇄 |
| W2 | 고민 ← 도움됨 ← 성분 | 직접, 1-hop |
| W3 | 고민 ← 악화 ← (작동원리\|성분) | 회피 필터 |
| W4 | 사용자 → 고민보유 → 고민 + (W1~W3) + 사용자 → 회피 → 성분(최종 제외) | 개인화 결합 |
| W5 | 성분 ↔ 성분 궁합 | 별도 경로. "A랑 B 같이 써도 돼?" 질의용 |

**규칙**

- 지식체인은 최대 3-hop(사용자를 포함하면 4 edge)까지만 허용한다. **무제한 10-hop 탐색은
  금지한다.**
- 신뢰성은 경로 길이가 아니라 근거 문서 수(확증 문서 개수)로 판단한다.
- 경로의 끝은 성분(추천), 악화(회피), 또는 궁합쌍(W5) 중 하나여야 한다.
- 관계 방향의 일관성을 지켜야 한다. 예: TREATS 뒤에 CONFLICTS를 이어붙여 추천 결론을 내는
  것은 금지한다.

## 6. Provenance(출처)와 신뢰도

> **요약**: 그래프의 모든 관계는 "어느 문서에서 나온 근거인지"를 반드시 달고 있어야 한다.
> DB에서 나온 확실한 사실과 LLM이 문서에서 추출한(틀릴 수 있는) 내용을 구분해서 표시한다.

**왜 필요한가**: LLM이 아티클에서 뽑은 관계는 틀릴(환각) 수 있다. origin으로 하드팩트(DB)와
구분하지 않으면 환각 관계가 진짜 사실과 같은 무게로 추천을 몰고 간다. source_doc_ids가 있어야
"이 관계를 몇 개 문서가 뒷받침하는지"로 신뢰도를 매길 수 있다.

모든 지식 엣지(ACHIEVES / ENABLES / TREATS / AGGRAVATES / HELPS / CONFLICTS)는 다음 프로퍼티를
필수로 가진다.

- `source_doc_ids`: 근거가 된 청크 doc_id (복수 가능)
- 원본 아티클 url
- `confidence`
- `origin`: `rdb` / `rule` / `llm_doc` 중 하나

하드팩트(DB에서 나온 사실)와 LLM 추출(환각 가능성이 있는 것)을 `origin`으로 구분한다. 신뢰도는
확증 문서 수로 랭킹을 매긴다.

## 7. 엔티티 해소(alias 레이어) — 왜 필요하고 어디까지 필요한가

> **요약**: 같은 성분을 어디서 말하든 하나의 노드로 모으는 게 목표다. 이건 "노드 키를
> 뭘로 하느냐(정체성)"와는 별개로, "한글 텍스트를 올바른 노드로 연결하느냐(해소)"의 문제다.

### (1) 같은 성분을 하나로 모으려면 세 층이 다 필요하다

| 층 | 하는 일 | 예 | 담당 |
|---|---|---|---|
| ① 행 중복 제거 | 같은 성분 중복 행 정리. 단 현재 데이터상 제품이 여러 id로 갈라진 경우 0건이라 필수가 아니며 ③ alias가 커버 | glycerin/glycerine(같은 Glycerin), lovage/levisticum_officinale(같은 러비지) | 선택적(현재 데이터상 불필요) |
| ② 노드 키 | 노드 키 = ingredient_id(언어중립·안정) | — | ③가 제품 연결 id로 보냄을 전제 |
| ③ 표면형 해소(alias) | 글/질의의 한글 표현을 그 노드로 연결 | BHA/살리실산 → 살리실릭애씨드 행 | 추출·질의 공유 |

이전에 "정체성은 ingredient_id로 끝"이라 한 건 부정확했다 — ②는 ③(alias가 표면형을
"제품에 연결된 id"로 보냄)을 전제로 성립한다. 같은 성분(예: 글리세린)이 두 id로 갈라져
있어도, 실제로는 제품이 한 id(예: 47개)에 몰려 있고 다른 id(0개)는 제품 연결이 없는 고아
행이라 질의·추출에 문제가 되지 않는다. (같은 name_ko라도 실제 다른 성분일 수 있어 — 예:
레몬 `lemon` vs `citrus_medica`=시트론 — 이런 경우는 애초에 병합 대상이 아니다.)

실데이터 확인: 중복 성분 28그룹 중 제품이 여러 id로 쪼개진 경우 0건(대부분 한 id에 제품이
몰려 있고, 나머지 중복 행은 제품 연결이 없어 무해). 따라서 INCI 기준 사전으로 행을 병합할
필요 없이, alias 맵이 각 한글·영문·관용어 표면형을 '제품에 연결된 id(없으면 name_en·intro
등 지식 있는 사전 행)'로 보내면 된다.

### (2) INCI 병합(①)과 관용어 alias(③)는 다르다 — BHA 예

- ①(INCI 병합)이 합치는 건 **같은 INCI의 표기 변형**뿐이다(glycerin/glycerine).
- **BHA와 살리실릭애씨드는 INCI가 다르다.** 애초에 "BHA"는 성분이 아니라 "베타하이드록시
  애씨드"라는 **클래스**이고, 살리실릭애씨드가 그 대표 멤버라 관용적으로 BHA=살리실릭애씨드로
  부르는 것뿐이다. DB에도 "BHA" 행은 없고(살리실릭애씨드 행만 존재), "BHA"는 아티클 텍스트에만
  나온다.
- 따라서 BHA→살리실릭애씨드는 ①(중복 제거)이 아니라 **③(관용어 alias, 도메인 지식 큐레이션)**
  으로 처리한다. AHA→글라이콜릭/락틱, 비타민C→아스코빅애씨드도 같은 ③.

### (3) alias 맵을 어디서·어떻게·어떤 상황에 쓰나

**어디서 — 쓰기 1곳 + 읽기 1곳(같은 맵 공유)**

- ① 트리플 추출 시(적재, A): LLM이 아티클에서 뽑은 표면 문자열("BHA","각질제거","여드름")을
  그래프에 저장하기 전에 노드 식별자로 변환. 안 하면 "BHA" 노드와 "살리실릭애씨드" 노드가
  갈라진다.
- ② 질의 해석 시(검색): 사용자의 "BHA","트러블"을 노드로 링킹해 순회 시작.
- 핵심: 두 지점이 같은 맵을 쓰므로 아티클의 "BHA"와 질문의 "BHA"가 같은 노드에서 만난다.

```
아티클 "BHA가 각질을 제거한다"          질문 "BHA 추천해줘"
        │ 쓰기측(추출)                          │ 읽기측(질의)
        ▼                                        ▼
              alias 맵: BHA → 살리실릭애씨드(id)
                          │
                          ▼
              같은 노드(살리실릭애씨드)에서 만남
```

**어떻게 — 룩업 캐스케이드**

1. alias 맵 정확 매칭 (BHA→639, 트러블→concern:acne, 필링→mechanism:각질제거)
2. 없으면 `ingredients.name_ko` 직접 매칭 (살리실릭애씨드→그 행; 표에 있는 이름은 맵에
   안 넣어도 됨)
3. 그래도 없으면 미해소 로그+skip(추출) / 되물음(질의)

alias 타깃 선정: 같은 성분의 여러 id 중 제품에 연결된 id 우선, 없으면 name_en·intro(지식)
있는 행.

구조는 공유 테이블 하나.

| surface(한글표현) | 대상 종류 | 대상 |
|---|---|---|
| BHA, 살리실산 | ingredient | (살리실릭애씨드 id) |
| 비타민C, 비타민씨 | ingredient | (아스코빅애씨드 id) |
| 트러블, 뾰루지 | concern | acne |
| 필링, 각질케어 | mechanism | 각질제거 |

**어떤 상황에서 결과가 갈리나**

- **확증(신뢰도)**: 아티클 A "BHA→각질제거", B "살리실릭애씨드→각질제거"를 맵이 같은 노드로
  보내면 같은 관계를 2문서가 뒷받침(confidence↑). 없으면 1+1로 흩어져 신뢰도가 낮게 잡힌다.
- **경로 히트**: 질문 "BHA 추천"이 맵 없이는 그래프의 살리실릭애씨드 노드를 못 찾아 결과
  0건. 맵이 있으면 연결돼 정상 추천.

### (4) 필요성은 노드마다 다르다 (범위 축소)

- **성분(Ingredient)**: `ingredients.name_ko`가 이미 있어 "레티놀"처럼 표에 있는 이름
  그대로면 직접 매칭으로 대부분 커버된다. alias는 관용어·약어·영문(BHA/AHA/비타민C)만
  보완한다 — 큰 사전을 미리 만들 필요 없이 매칭 실패분만 점진 추가.
- **고민(Concern)**: 여드름/트러블/뾰루지→acne 같은 작은 목록이고 이미 CONCERN_RULES에
  대부분 있다.
- **작동원리(Mechanism)**: ⭐진짜 이유. RDB에 대응 표가 없고 자유 텍스트에서 뽑혀 표기
  변형이 크다 — 여기가 alias의 진짜 주 대상이다.

### (5) 제외 규칙 큐레이션 / 고민 흡수 / canonical_key 위치

**제외 규칙 큐레이션**

- 자극 ≠ 민감 (자극은 유발 문맥으로도 쓰인다)
- 노화 ⊋ 주름 (노화가 상위 개념)
- "생기", "모이스처(제품명)", "피지(작동원리↔고민 동음이의)"는 순수 키워드 매칭 시 노이즈가
  된다.

**고민 흡수 규칙**

- 색소침착 · 피부톤 · 미백 → 칙칙함
- 탄력 → 주름

**canonical_key 위치**

노드 정체성 키에서는 내려놓는다. 행 병합이 현재 불필요하므로 canonical_key는 A의 크롤
단계에서 중복 정리 시 참고용일 뿐 필수는 아니다. INCI 기준 사전이 없어도 alias로 해결된다.

**범위 메시지**

"거대한 한글 정규화 시스템"이 아니라, 실질적으로 필요한 건 ③작동원리(Mechanism) 매핑 +
성분 관용어·영문 표면형 몇 개다.

## 8. 추출 파이프라인

> **요약**: 문서를 통째로 넣지 않고 잘게 나눠서(청킹) 처리한다. 그중 "참고문헌 목록"처럼
> 관계 추출에 쓸모없는 조각은 미리 걸러낸다. LLM 호출은 무료 쿼터 제약이 있어서, 바뀐 문서만
> 골라 처리하는 방식이 필요하다.

**왜 필요한가**: 참고문헌 청크(33%)는 관계 정보가 없어 추출해봤자 쓰레기 트리플만 나오고
쿼터만 낭비하므로 제외한다. 증분 추출을 하는 이유는, 무료 Gemini 쿼터 한계상 306청크를 매번
다시 분석하면 쿼터가 금방 소진되므로, 바뀐 문서만 처리하기 위해서다.

- **청킹 도입됨**: `source_meta`에 `is_chunk`/`chunk_index`가 있다. 147개 원본 문서가
  306개 청크로 나뉜다. 트리플 추출은 청크 단위로 하고, provenance는 청크 doc_id와 원본
  url을 함께 기록한다.
- **추출 입력 정제**: 참고문헌 청크는 제목이 "[아티클제목 - 참고 자료]" 형태의 **별도
  청크**다(실데이터 306청크 중 100청크가 참고문헌, 전부 제목 레벨이고 본문에 섞인 경우
  0건 확인). 따라서 제목/섹션이 참고문헌인 청크를 **통째로 제외**하면 본문 손실 없이
  안전하게 걸러진다.
- **비용/쿼터**: LLM 추출은 무료 Gemini 쿼터 제약이 있다. 문서 변경 감지를 위한 해시 기반
  **증분 추출**과 배치 처리가 필요하다(매번 전체 문서를 재분석하는 방식은 금지).

## 9. 작동원리(Mechanism) 노드 도출 — 아티클 데이터 근거

**왜 필요한가**: 개념을 임의로 정하면 아티클의 실제 표현과 안 맞아 추출이 그 노드를 못 채운다.

## 10. 그래프 빌드 로직 변경 (현 `knowledge_populate.py` 대비)

> **요약**: 지금의 그래프 생성 코드 중 절반(개인화 관련 부분)은 그대로 두고, 나머지 절반
> (전역 지식 부분)은 "아티클에서 추출한 관계를 로드하는 방식"으로 새로 만든다.

**왜 필요한가**: 그래프의 "의미"가 바뀌기 때문이다 — 전역 지식이 intro 키워드 매칭이 아니라
아티클 근거 있는 트리플로 만들어지고, 성분 키가 ingredient_id로 바뀌고, 제품이 그래프에서
빠진다. 그래서 전역 지식 생성 부분은 새로 짜야 한다. 개인 엣지(memories 투영)는 의미가 안
바뀌어 유지한다.

| 항목 | 처리 |
|---|---|
| Concern 노드 생성 | 유지 |
| Ingredient 노드(canonical_key MERGE) | 재작성 — `ingredient_id` 키로, 한글키 skip 로직 제거 |
| Product 노드 + CONTAINS 엣지 | 삭제 — 제품은 RDB JOIN으로 대체 |
| TREATS/AGGRAVATES(intro 키워드 기반) | 대체 — 아티클 트리플 추출 + provenance |
| HELPS/CONFLICTS(하드코딩 PARTNERSHIPS) | 대체 — 아티클 추출(W5) |
| Mechanism 노드 + ACHIEVES/ENABLES | 신규 |
| User + AVOIDS/PREFERS/HAS_CONCERN(memories 투영) | 대체로 유지 — AVOIDS/PREFERS는 ingredient_id를 가리키고, HAS_CONCERN 한글 라벨 해소 로직도 유지 |

요지: 전역 지식 절반은 "추출 트리플 로더"로 재작성/대체하고, 개인 엣지 절반은 그대로
유지한다.

## 11. A에게 전달할 플래그 & 지시

> **요약**: 지금 당장 확인이 필요한 문제 3가지와, 새 그래프 구현 시 반드시 지켜야 할 지시
> 5가지를 정리한다.

### 즉시 확인 필요 플래그 3건

1. **DB 초기화 스크립트 충돌**: `db/initdb/06-restore.sh`가 덤프에 포함된 AGE 그래프
   카탈로그와 `03-graph-ontology.sql`을 충돌시킨다("User already exists" 등 37개 에러
   발생 → `pg_restore` non-zero 종료 → 볼륨 재생성 시 컨테이너 init 실패). RDB 데이터는
   그 전에 복원되지만 수동 재시작이 필요한 상태다. **해법**: 덤프에서 AGE 그래프 카탈로그를
   제외한다. A가 pg_dump 시 public RDB 테이블만 뜨거나(예: `pg_dump --data-only -t
   public.products -t public.ingredients -t public.documents -t public.product_ingredients
   -t public.memories`), 또는 `--exclude-schema=ag_catalog --exclude-schema=skinmate`.
   그러면 03-graph-ontology.sql이 그래프를 깨끗이 만들고, 그래프 데이터는 이후 적재
   단계가 채운다.
   **왜 문제인가**: 볼륨을 새로 만들 때마다 init이 실패 → 개발 환경 재현·CI가 막힌다.
2. **`ingredients.canonical_key` 한글 오염**: 새 크롤러 데이터에서 `canonical_key`가
   한글인 행이 417/1045(40%)이며, 전부 `name_en`이 없다(중복쌍은 1개뿐 — 옛 중복 문제는
   사실상 해결된 상태). 근본 원인은 `ingest/normalize.py`의 한글 폴백 로직과 사전
   커버리지 부족이다. canonical_key가 한글이어도 그래프에는 문제가 없다(노드 키는
   ingredient_id). 표면형은 alias 맵에 한글·영문·관용어를 '제품 연결 id'로 매핑해
   해결한다. **INCI 기준 사전을 새로 만들 필요 없음** — 실데이터상 중복 28그룹 중 제품이
   여러 id로 갈라진 경우가 0건이라 행 병합도 불필요하다.
   **왜 문제인가**: 성분 표기가 파편화되면 §7대로 확증 카운트·추출 정확도가 떨어진다.
3. **참고문헌 청크 오염**: `documents` 306청크 중 33%(100청크)가 참고문헌 목록이다.
   **해법**: 제목이 '참고 자료'인 별도 청크(100개)를 추출 입력에서 통째로 제외한다. 본문
   섞임이 없어 안전(§8).
   **왜 문제인가**: 참고문헌 청크를 안 거르면 §8대로 쓰레기 트리플·쿼터 낭비가 생긴다.

### 5대 지시

1. 메타패스 화이트리스트 + hop 상한을 지킬 것(무제한 10-hop 탐색 금지).
2. 제품-성분 CONTAINS는 그래프에서 제거하고 RDB JOIN으로 대체한다. 성분 노드는
   `ingredient_id` 키를 사용한다.
3. 개념(작동원리/Mechanism) 노드는 아티클에서 도출한다(임의 정의 금지).
4. 엣지에는 `source_doc_ids`/`origin`/`confidence`를 필수로 부여한다(하드팩트와
   LLM 추출을 구분).
5. 해시 기반 증분 추출과 배치 처리로 LLM 쿼터를 최적화한다.

## 12. 역할 분담과 단계

> **요약**: 이 설계 전체를 A 혼자 다 하는 게 아니다. 작업은 세 갈래로 나뉘고, 한꺼번에 다 만들기보다 얇은 슬라이스로 먼저 검증한 뒤 확장한다.

### 12.1 작업은 세 갈래로 나뉜다

| 트랙 | 하는 일 | 관련 섹션 | 자연스러운 담당 |
|---|---|---|---|
| **A. 생산 측** (아티클 → 채워진 그래프) | 크롤러·정규화, 트리플 추출(LLM), 그래프 빌드 로더, concepts 테이블 | §6·§8·§9·§10 + §11 플래그 | A 중심(ingest·knowledge_populate 소유) |
| **B. 소비 측** (질문 → 답변) | 질의 엔티티 링킹, 메타패스 순회(traverse 재작성), 검색 융합(retrieve), 제품 RDB JOIN | §2·§5 | 질의·검색 담당 |
| **C. 공유 계약** (양측 인터페이스) | 온톨로지(노드/관계), alias 맵, provenance 스키마, 메타패스 화이트리스트 | §3·§4·§5·§6·§7 | A와 합의 필수 |

핵심: **C(공유 계약)를 먼저 합의**해야 A(생산)와 소비 측이 각자 병렬로 진행할 수 있다. 이 문서가 바로 그 C를 정하는 작업이다. 최종 담당 배정은 팀이 합의한다(큰 변경은 A와 합의).

### 12.2 한꺼번에 다 하지 않는다 — 얇은 슬라이스 → 확장

- **Phase 0 (합의)**: 이 문서로 공유 계약(온톨로지·화이트리스트·alias·provenance)을 확정한다.
- **Phase 1 (얇은 수직 슬라이스, 검증용)**: 고민 1개(예: 여드름) + **손으로 검증한 트리플 몇 개**(예: BHA→각질제거→여드름) + 성분 ingredient_id 키 + 메타패스 **W1 하나만** 순회 → 근거 아티클 회수 → 제품 RDB JOIN. LLM 추출·전체 alias 맵 없이 "근거 있는 추천" 흐름이 실제로 도는지 먼저 증명한다. 목적: 큰 파이프라인을 짓기 전에 아키텍처가 맞는지 확인하고 리스크를 줄인다.
- **Phase 2 (생산 파이프라인 구축, A)**: LLM 트리플 추출(source_doc_ids·origin·confidence, 해시 증분·배치), 참고문헌 청크 필터, alias 맵 확장, Mechanism 노드 도출, 그래프 빌드 로더.
- **Phase 3 (확장·검증)**: 나머지 메타패스(W1'·W2·W3·W4·W5), 신뢰도(확증 문서 수) 랭킹, 성능(hop 제한·경로 폭발 방어) E2E 검증.
